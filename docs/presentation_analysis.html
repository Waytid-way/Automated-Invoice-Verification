01 / SCALING ANALYSIS

ถ้าเอกสารเพิ่ม 10 เท่า?

วิเคราะห์ผลกระทบเมื่อปริมาณงานเพิ่มจาก 20 → 200 ใบ/วัน

📊 ปัจจุบัน (20 ใบ/วัน)

⏱OCR Processing: ~60 นาที (3.1s × 20)
🧠LLM Processing: ~60 นาที
👤Staff Review: ~40 นาที (2 นาที/ใบ)
📁Storage: JSON file เพียงพอ
🌐API: Rate limit ไม่เป็นปัญหา

📈 10x Scale (200 ใบ/วัน)

⏱OCR: ~600 นาที (10+ ชม.)
🧠LLM: ~600 นาที
👤Staff Review: ~400 นาที (6-7 ชม.) 🔴
📁Storage: JSON เริ่มช้า, ไม่รองรับ Concurrent
🌐API: OCR rate limit ตึงมาก

📌 สรุป: Computer ปรับได้ แต่ "คน" คือ Hard Cap

Computer side: parallelize + เปลี่ยน DB → ลดเวลา processing ได้เหลือ ~30-60 นาที
Human side: staff 1 คน ดูได้ ~100 ใบ/วัน — ถ้า 200 ใบ ต้องเพิ่มคน หรือ auto-approve

02 / BOTTLENECK ANALYSIS

คอขวดหลักคืออะไร?

วิเคราะห์ 4 Bottlenecks ที่เกิดขึ้นเมื่อ Scale

🔴 #1 Staff Review

คน 1 คน ดูได้ ~100 ใบ/วัน → 200 ใบ ต้อง 2 คน หรือ auto-approve

HARD CAP คอขวดหลัก

🔵 #2 OCR Rate Limit

Typhoon OCR limit: 20 req/min → 200 ใบ รอ OCR queue ~600 นาที

# แก้ได้ด้วย Parallel with ThreadPoolExecutor(max_workers=5) as ex: results = ex.map(ocr, batch)
ปรับได้

🟣 #3 JSON Storage

JSON file ไม่รองรับ concurrent writes → หลาย process เขียนพร้อมกัน = Data Race

เปลี่ยน DB

🟢 #4 LLM Latency

200 req/min limit แต่ละ call ~2-5 วินาที → 200 ใบ = ~400-1,000 วินาที

Async + Batch

⚠️ คอขวดหลัก: Staff Review — ไม่ใช่ Computer

Computer ปรับได้หมด (parallelize OCR + async LLM + PostgreSQL) แต่ คนดูได้วันละ ~100 ใบ — นี่คือขีดจำกัดที่แก้ไม่ได้โดยไม่เพิ่มคน

03 / CONSTRAINTS

ข้อจำกัดของระบบ

Constraint คือข้อจำกัดที่ไม่สามารถเปลี่ยนแปลงได้

Constraint รายละเอียด ประเภท
OCR Rate Limit Typhoon OCR จำกัด 20 req/min — hard limit จาก API External API
Staff Capacity คน 1 คน ดูได้ ~100 ใบ/วัน (2 นาที/ใบ) Human Resource
JSON Concurrent Writes JSON file ไม่รองรับหลาย process เขียนพร้อมกัน Storage

Image Quality Dependency ภาพเบลอ/มืด → OCR อ่านผิดได้ง่าย Input Quality

KEY INSIGHTS

💡 บทเรียนสำคัญ

1. System Design ไม่ใช่แค่โค้ด
ต้องคิดถึง Human constraints, API limits, และ business realities

2. Hard Cap มาจากคน ไม่ใช่เครื่อง
Computer scale ได้ แต่คนมีขีดจำกัด — ออกแบบระบบให้คนใช้งานได้จริง

3. Trade-off มีทุกที่
ไม่มี "ถูก" ตลอด — ทุก decision มี cost ต้องชั่งน้ำหนักก่อนเลือก

4. รู้จัก Constraints ของตัวเอง
API Rate Limit, Storage, Staff — นี่คือขอบเขตที่ต้องออกแบบภายใน

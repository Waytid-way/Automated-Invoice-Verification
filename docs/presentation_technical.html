01 / WHY MULTI-AGENT

ทำไมต้องแบ่ง Agent?

แทนที่จะใช้ Single Agent ทำทุกอย่าง — ทำไมต้องแยกให้แต่ละ Agent มีหน้าที่เฉพาะ

❌ Single Agent Problem

⚠️ LLM เดียวทำทุกอย่าง → โอเวอร์โหลด
⚠️ ไม่มี Error Recovery ที่ชัดเจน
⚠️ Debug ยาก — ไม่รู้ว่าขั้นตอนไหนผิด
⚠️ OCR Model กับ LLM Model ใช้คนละตัว แต่รวมกัน?
⚠️ ยากต่อการ Scale — อยากเปลี่ยน OCR แต่ต้องแก้ทั้งระบบ

✅ Swarm Multi-Agent Solution

✓ แต่ละ Agent มีหน้าที่เฉพาะ — โฟกัสได้ดี
✓ Agent ส่งผลต่อให้กัน (Handoff) — Flow ชัดเจน
✓ Error ที่ Agent ไหน → Debug ได้ง่าย
✓ OCR กับ LLM Model แยกกันตามจริง
✓ Scale ได้ — เปลี่ยน Agent ได้โดยไม่กระทบตัวอื่น

02 / AGENT DETAILS

แต่ละ Agent ทำอะไร?

รายละเอียดหน้าที่ของแต่ละ Agent ใน Swarm Architecture

📄 OCR Agent

  • รับ Input: ไฟล์ PDF / JPG / PNG
  • ใช้ Typhoon OCR (3B Model)
  • สกัด Raw Text ออกมา
  • Retry: 3 ครั้ง
  • Fallback: LOCAL Ollama

🧠 LLM Extraction Agent

  • รับ Input: Raw Text จาก OCR
  • ใช้ Typhoon LLM (30B Model)
  • สกัดเป็น Structured JSON
  • มี Few-shot Prompt กำหนด Schema
  • Thai VAT Auto-calculation

🔍 Duplicate Detection

  • Layer 1: Exact Match (Invoice Number)
  • Layer 2: Fingerprint (Vendor+Amount+Date)
  • Layer 3: Fuzzy Match (Similarity ≥ 0.85)
  • Return: DuplicateResult

⚖️ PO Matching Agent

  • 2-Way Matching: PO Number + Amount
  • Status: MATCHED / AMOUNT_MISMATCH / NOT_FOUND
  • คำนวณ Diff ระหว่าง Invoice กับ PO
  • Return: POMatchResult

🛡️ Error Handler Agent

  • Severity: LOW → MEDIUM → HIGH → CRITICAL
  • Retry Logic ทุก Agent
  • Rate Limit Cooldown (10 วินาที)
  • Fallback: Cloud → LOCAL

💡 Key Insight: Sequential Handoff

แต่ละ Agent ทำงานเสร็จแล้วส่งผลลัพธ์ให้ Agent ถัดไป ข้อดีคือ Flow ชัดเจน Debug ง่าย แต่ถ้า Batch ใหญ่ต้องรอขั้นตอนเดียวเสร็จก่อน

03 / HUMAN-IN-THE-LOOP

ทำไมต้องมี HITL?

แม้ AI จะทำงานได้ดี แต่ Human Checkpoint ยังจำเป็น — ดูเหตุผล

🤖 AI อาจผิดพลาด

LLM อาจ "หลอน" (Hallucination) — สกัด 1,000 บาท ว่า 10,000 บาท ซึ่งกระทบทางการเงินโดยตรง ตัวอย่าง: อ่านเลข 1 เป็นเลข 7

HIGH RISK

⚖️ ความรับผิดชอบทางการเงิน

การอนุมัติจ่ายเงินต้องมี ชื่อจริง + ลายเซ็น AI ไม่สามารถรับผิดทางกฎหมายได้ ต้องมีคนรับผิดชอบจริง

COMPLIANCE

📋 Compliance / Audit

มาตรฐานทางการเงินกำหนดให้ทุกการอนุมัติต้องมี หลักฐานประจำตัวผู้อนุมัติ + วันเวลา บันทึกใน Audit Log ต้องมีความถูกต้องสมบูรณ์

REQUIRED

⚠️ False Positive / Negative

Duplicate Detection อาจ False Positive (บอกซ้ำทั้งที่ไม่ซ้ำ) หรือ False Negative (บอกไม่ซ้ำทั้งที่ซ้ำ) — ต้องมีคนตัดสินใจสุดท้าย

04 / RATE LIMITING

กลยุทธ์ Rate Limiting

ทำอย่างไรไม่ให้โดน Typhoon API Ban — เมื่อเจอ 429 ต้องทำยังไง

📊 Typhoon Cloud Limits

  • OCR: 20 req/min
  • LLM: 200 req/min

OCR คือ Bottleneck — ต้องระวังกว่า LLM

✅ Our Solution: 3.1s Delay

for invoice in batch: ocr_result = typhoon_ocr(image) sleep(3.1) # รอก่อนใบต่อไป # 3.1 × 20 = 62 วินาที ต่อ 20 calls # ปลอดภัยไม่โดน Ban ✅

🔄 เมื่อเจอ 429 Error

  • Step 1: Cooldown 10 วินาที
  • Step 2: Retry ครั้งที่ 1
  • Step 3: ถ้ายัง 429 อีก → ข้ามใบนั้น
  • Step 4: Log Error + แจ้ง Staff

🔀 Fallback Strategy

  • Cloud API Fail → Fallback ไป LOCAL Ollama
  • LOCAL ฟรี + ไม่มี Rate Limit
  • Trade-off: เครื่องต้องมี GPU แรงพอ

05 / DUAL MODE

LOCAL vs CLOUD

ระบบรองรับ 2 โหมด — User เลือกได้ตามสถานการณ์

Criteria LOCAL (Ollama) API (Typhoon Cloud)
💰 ค่าใช้จ่าย ✅ ฟรี (ต้องมี GPU) ⚠️ มีค่าใช้จ่ายตามการใช้งาน
🌐 Internet ✅ ไม่ต้องใช้เน็ต ❌ ต้องมีเน็ตตลอด
🔒 ความเป็นส่วนตัว ✅ ข้อมูลไม่ออกนอกเครื่อง ⚠️ ข้อมูลส่งไป Cloud
⚡ Performance ✅ Latency ต่ำ (ไม่มีเน็ต) ⚠️ Latency ขึ้นกับเน็ต
💻 Hardware ❌ ต้องมี GPU แรงพอ ✅ ใช้งานได้ทุกเครื่อง
🔧 Setup ❌ ต้องติดตั้ง Ollama ✅ พร้อมใช้ทันที
📦 Model Size ⚠️ ขึ้นกับ VRAM ✅ เข้าถึง 30B Model เต็ม

⚙️ วิธีเลือกโหมด

ใน Streamlit Sidebar → เลือก "LOCAL (Ollama)" ถ้ามี GPU แรงพอและต้องการความเป็นส่วนตัว หรือเลือก "API (Typhoon Cloud)" ถ้าต้องการความสะดวกและเข้าถึงโมเดลใหญ่กว่า

06 / SUMMARY

สรุป Technical Decisions

5 จุดสำคัญที่ต้องจำ — พร้อมเหตุผล

1 Swarm Multi-Agent

แต่ละ Agent มีหน้าที่เฉพาะ → Debug ง่าย, Scale ง่าย, เปลี่ยน Model ได้โดยไม่กระทบตัวอื่น

2 HITL Checkpoints

Staff ตรวจทุกใบก่อน Approve → ป้องกัน AI Error ที่กระทบทางการเงิน + มี Audit Trail ตามกฎหมาย

3 Rate Limiting

3.1s delay สำหรับ OCR + Cooldown 10s เมื่อเจอ 429 → ไม่โดน Ban แม้ Batch ใหญ่

4 Dual-Mode

LOCAL/CLOUD เลือกได้ตามสถานการณ์ → LOCAL ฟรี+เป็นส่วนตัว, CLOUD สะดวก+เข้าถึงโมเดลใหญ่

5 Fallback Chain

Cloud → LOCAL Ollama → Mark Failed → Log Error → Alert Staff → ไม่มีใบหายโดยไม่รู้

KEY TAKEAWAYS

💡 บทเรียนสำคัญ

1. AI ช่วยลดเวลา
แต่ไม่ใช่ทดแทนมนุษย์ — HITL จำเป็นเสมอสำหรับงานทางการเงิน

2. Swarm ไม่ใช่ทุกอย่างต้อง Parallel
Sequential Handoff ก็เพียงพอสำหรับงานที่ต้องรอขั้นตอนก่อน

3. Rate Limit = Survival
ระบบที่ดีต้องรู้ว่าขีดจำกัดของ API อยู่ตรงไหน และมี Backup Plan

4. Thai Language Support
Typhoon OCR/LLM รองรับภาษาไทยดีมาก — เหมาะสำหรับ Invoice ไทย

07 / SCALING

ถ้าเอกสารเพิ่ม 10 เท่า?

วิเคราะห์ Bottleneck + วางแผน Scale ระบบ

📊 Current State (20 ใบ/วัน)

⏱ Processing: ~60-100 นาที (OCR 3.1s × 20)
👤 Staff Review: ~40 นาที (2 นาที/ใบ × 20)
📁 Storage: JSON file เพียงพอ
🌐 API: Rate limit ไม่เป็นปัญหา

📈 10x Scale (200 ใบ/วัน)

⏱ Processing: ~600-1,000 นาที (10+ ชม. เลย!)
👤 Staff Review: ~400 นาที (6-7 ชม. ต่อวัน)
📁 Storage: JSON เริ่มช้า, ไม่รองรับ Concurrent
🌐 API: OCR rate limit ตึงมาก (200/20 = 10 เท่า)
BOTTLENECK ANALYSIS

🔴 Bottleneck #1: OCR Rate Limit

OCR limit: 20 req/min — ถ้า 200 ใบต้องรอ ~600 นาที ขึ้นอยู่กับ rate limit

# Current: Sequential for invoice in batch: result = ocr(invoice) # 3.1s delay # 200 ใบ = 620 วินาที = 10+ นาที แค่ OCR # 10x: Parallel OCR พร้อมกัน with ThreadPoolExecutor(max_workers=5) as ex: results = ex.map(ocr, batch) # ลดเวลาลง 5 เท่า

🟠 Bottleneck #2: JSON File Storage

JSON file ไม่รองรับ Concurrent writes — ถ้าหลาย process เขียนพร้อมกันจะ Data Race

  • ต้องเปลี่ยน → PostgreSQL หรือ SQLite
  • รองรับ ACID + Concurrent Access
  • Index บน Invoice Number, PO Number

🟣 Bottleneck #3: Staff Review

200 ใบ × 2 นาที = ~400 นาที (6-7 ชม.) — พนักงาน 1 คนรับไม่ไหว

  • เพิ่ม Staff Reviewer หลายคน
  • Triage System: High-risk ให้ Senior ดูก่อน
  • Auto-prioritize: Mismatch → Senior, Matched → Junior

🔵 Bottleneck #4: LLM Latency

LLM 200 req/min — แต่ละ request ~2-5 วินาที → 200 ใบ = ~400-1,000 วินาที

  • Parallel LLM calls ด้วย AsyncIO
  • Caching: ถ้า Invoice คล้ายกัน → ใช้ cached result
  • Batching: ส่งหลายใบใน prompt เดียว (few-shot)
SOLUTION ROADMAP
Phase Action Impact Effort
Quick Win Parallel Batch Processing (ThreadPoolExecutor) ↓ 50-70% เวลา OCR ต่ำ
Medium PostgreSQL + Redis Cache รองรับ Concurrent + Fast lookup ปานกลาง
Major AsyncIO LLM Calls + Request Batching ↓ 40% LLM latency สูง
Future Message Queue (RabbitMQ / Kafka) รองรับ 1000+ ใบ/วัน สูงมาก

📌 สรุป: สิ่งที่ต้องทำเมื่อ Scale 10x

🔀
Parallelize OCR

ThreadPoolExecutor + Semaphore คุม rate limit

🗄️
เปลี่ยน Database

JSON → PostgreSQL รองรับ Concurrent

👥

Triage + Priority Queue สำหรับ Staff

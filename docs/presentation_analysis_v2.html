03 — Constraints

ข้อจำกัด ของระบบ

Constraint คือข้อจำกัดที่เปลี่ยนแปลงไม่ได้ — ต้องออกแบบภายในขอบเขตนี้

#1

OCR Rate Limit — External API

Typhoon OCR จำกัด 20 req/min — hard limit จาก Typhoon API ไม่สามารถเปลี่ยนได้

External API
#2

Staff Capacity — Human Resource

คน 1 คน ดูได้ ~100 ใบ/วัน (2 นาที/ใบ) — ขีดจำกัดทางกายภาพ

Human Resource
#3

LLM Hallucination — AI Limitation

AI อาจสกัดข้อมูลผิด — ต้องมี HITL check ทุกใบ ไม่สามารถ auto-approve 100%

AI Limitation
#4

Image Quality Dependency

ภาพเบลอ/มืด/ถ่ายเอียง → OCR อ่านผิดได้ง่าย — input quality ควบคุมไม่ได้

Input Quality
🚫
2 Constraints ที่แก้ไม่ได้
OCR Rate Limit (External) + Staff Capacity (Human) — ต้องออกแบบระบบภายในขอบเขตนี้
04 — Trade-offs

การชั่งน้ำหนัก ในการออกแบบ

ทุกการตัดสินใจมี Trade-off — ไม่มีคำตอบที่ "ถูก" ทั้งหมด

⚖️
Speed vs Safety
Auto-Approve
เร็ว แต่เสี่ยง
Human Approve

✅ Human Approve — ยอมช้าแต่ป้องกันความผิดพลาดทางการเงิน
⚖️
Cost vs Speed
LOCAL Ollama
ฟรี แต่ต้อง GPU
Cloud API
เร็ว แต่มีค่าใช้จ่าย
✅ Dual Mode — ให้ user เลือกตามสถานการณ์
⚖️
Simplicity vs Scale
JSON File

PostgreSQL

✅ JSON ก่อน — เริ่มง่าย ขยายทีหลังเมื่อจำเป็น

Constraints

Trade-offs

ทุก decision มี trade-off

💡 บทเรียนสำคัญ

1
System Design ไม่ใช่แค่โค้ด
ต้องคิดถึง Human constraints, API limits, และ business realities
2
Hard Cap มาจากคน ไม่ใช่เครื่อง
Computer scale ได้ แต่คนมีขีดจำกัด
3
Trade-off มีทุกที่
ไม่มี "ถูก" ตลอด — ทุก decision มี cost
4
รู้จัก Constraints ของตัวเอง
นี่คือขอบเขตที่ต้องออกแบบภายใน

# การวิเคราะห์กระบวนการทำงาน (Workflow Analysis)

## 1. กระบวนการทำงานปัจจุบัน (Manual Workflow)

กระบวนการตรวจสอบใบแจ้งหนี้แบบแมนนวลในปัจจุบัน มีขั้นตอนทั้งหมด 10 ขั้นตอน ใช้เวลาประมาณ 20-40 นาทีต่อใบแจ้งหนี้ 1 ฉบับ

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                        MANUAL INVOICE VERIFICATION WORKFLOW                       │
│                              (20-40 นาที/ใบแจ้งหนี้)                              │
└─────────────────────────────────────────────────────────────────────────────────┘

  ขั้นตอนที่ 1              ขั้นตอนที่ 2              ขั้นตอนที่ 3
┌──────────────┐      ┌──────────────┐         ┌──────────────┐
│   ได้รับ      │      │   เปิดเอกสาร   │         │  ค้นหาใบสั่งซื้อ  │
│  ใบแจ้งหนี้   │─────▶│  แต่ละฉบับ    │────────▶│   (Purchase   │
│  จากผู้ขาย    │      │   (PDF/Image) │         │   Order)     │
└──────────────┘      └──────────────┘         └──────────────┘
                                                    │
                                                    ▼
  ขั้นตอนที่ 8              ขั้นตอนที่ 7              ขั้นตอนที่ 4
┌──────────────┐      ┌──────────────┐         ┌──────────────┐
│   บันทึกข้อมูล  │      │   อนุมัติ      │         │   เปรียบเทียบ  │
│   เข้าระบบ    │◀─────│   อนุมัติ     │────────▶│   ข้อมูลแบบ   │
│    ERP        │      │   ด้วยตนเอง   │         │   แมนนวล     │
└──────────────┘      └──────────────┘         └──────────────┘
       │                                              │
       │                                              ▼
       │               ขั้นตอนที่ 6              ขั้นตอนที่ 5
       │         ┌──────────────┐         ┌──────────────┐
       │         │   ตรวจสอบ     │         │   ตรวจสอบ     │
       └─────────│   รายการที่   │◀────────│   รายการที่   │
                 │   ผ่านมาแล้ว   │         │   ซ้ำซ้อน     │
                 │  (Paid List)  │         │              │
                 └──────────────┘         └──────────────┘
```

### รายละเอียดขั้นตอน (10 ขั้นตอน)

| ขั้นตอน | รายละเอียด | ผู้รับผิดชอบ |
|---------|-----------|-------------|
| 1 | ได้รับใบแจ้งหนี้จากผู้ขายทางอีเมล/ไปรษณีย์/แฟกซ์ | ฝ่ายจัดซื้อ |
| 2 | เปิดเอกสารแต่ละฉบับ (PDF, รูปภาพ, กระดาษ) | พนักงานบัญชี |
| 3 | ค้นหาใบสั่งซื้อ (Purchase Order) ที่เกี่ยวข้องในระบบ | พนักงานบัญชี |
| 4 | เปรียบเทียบข้อมูลใบแจ้งหนี้กับใบสั่งซื้อด้วยตนเอง (ราคา, จำนวน, รายละเอียดสินค้า) | พนักงานบัญชี |
| 5 | ตรวจสอบรายการที่ซ้ำซ้อนในสมุดบันทึก/Excel | พนักงานบัญชี |
| 6 | ตรวจสอบรายการที่ชำระเงินแล้ว (Paid List) เพื่อป้องกันการเรียกเก็บซ้ำ | พนักงานบัญชี |
| 7 | อนุมัติการตรวจสอบด้วยลายมือชื่อหรือระบบอีเมล | หัวหน้าฝ่ายบัญชี |
| 8 | บันทึกข้อมูลเข้าระบบ ERP ด้วยตนเอง | พนักงานบัญชี |
| 9 | ตรวจสอบความถูกต้องของข้อมูลใน ERP อีกครั้ง | หัวหน้าฝ่ายบัญชี |
| 10 | จัดเก็บเอกสารต้นฉบับเข้าแฟ้ม | พนักงานบัญชี |

## 2. ปัญหาและผลกระทบ (Pain Points)

| # | ปัญหา | ผลกระทบ | ความรุนแรง |
|---|-------|---------|-----------|
| 1 | **ใช้เวลานาน 20-40 นาที/ใบแจ้งหนี้** - กระบวนการทั้งหมดต้องทำด้วยมือทุกขั้นตอน | ค่าใช้จ่ายด้านบุคลากรสูง ประสิทธิภาพต่ำ | สูงมาก |
| 2 | **เอกสารซ้ำซ้อนไม่ถูกตรวจพบ** - ไม่มีระบบตรวจจับอัตโนมัติ ต้องพึ่งพาความจำของพนักงาน | จ่ายเงินซ้ำ สูญเสียรายได้บริษัท | สูงมาก |
| 3 | **ข้อผิดพลาดในการคีย์ข้อมูล** - การคัดลอกข้อมูลจากเอกสารเข้าระบบด้วยมือ | ข้อมูลไม่ตรงกับต้นฉบับ ต้องตรวจสอบซ้ำ | สูง |
| 4 | **ค้นหาใบสั่งซื้อยาก** - ไม่มีระบบจับคู่อัตโนมัติ ต้องค้นหาด้วยตนเอง | เสียเวลาค้นหา หาข้อมูลผิดพลาด | ปานกลาง |
| 5 | **ไม่มี Audit Trail** - การอนุมัติด้วยลายมือชื่อ/อีเมล ไม่สามารถตรวจสอบย้อนกลับได้ | ขาดความโปร่งใส ความเสี่ยงด้านการกำกับดูแล | ปานกลาง |
| 6 | **ไม่มี Dashboard สถานะ** - ไม่ทราบว่าใบแจ้งหนี้อยู่ขั้นตอนใด ติดขัดตรงไหน | ตอบสนองลูกค้า/ผู้ขายช้า ขาดการวางแผนการจ่ายเงิน | ปานกลาง |

## 3. กระบวนการทำงานใหม่ที่ใช้ AI อัตโนมัติ (AI-Automated Workflow)

```
┌──────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┐
│                                   AI-AUTOMATED INVOICE VERIFICATION WORKFLOW                                          │
│                                    (2-5 นาที/ใบแจ้งหนี้ | ลดเวลา 80-90%)                                              │
└──────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┘

         ┌──────────┐
         │ Upload   │  ◀─── อัปโหลดไฟล์ทีละไฟล์หรือทีละแพ็ตช์ (PDF, รูปภาพ, ZIP)
         │  Batch   │
         └────┬─────┘
              ▼
         ┌──────────┐
         │   OCR    │  ◀─── Typhoon OCR: แปลงเอกสารภาพเป็นข้อความ (NIA Image + Thai)
         │ (Typhoon)│      รองรับ: ภาพถ่าย, PDF สแกน, ใบเสร็จเสียบเป็นภาพ
         └────┬─────┘
              ▼
         ┌──────────────────┐
         │   LLM JSON       │  ◀─── LLM Extraction: ดึงข้อมูลโครงสร้างจากข้อความ OCR
         │   Extraction     │      ผลลัพธ์: {invoice_no, date, vendor, items[], total}
         └────────┬─────────┘
                  ▼
         ┌──────────────────┐
         │   Duplicate      │  ◀─── การตรวจจับซ้ำ 3 ชั้น (3-Layer):
         │   Detection      │      ชั้นที่ 1: Exact Match (เลขที่ใบแจ้งหนี้ + ผู้ขาย)
         │   (3 Layers)     │      ชั้นที่ 2: Fingerprint Match (วันที่ + จำนวนเงิน + VAT)
         │                   │      ชั้นที่ 3: Fuzzy Match (ชื่อผู้ขายคล้าย + ยอดใกล้เคียง)
         └────────┬─────────┘
                  ▼
         ┌──────────────────┐
         │   PO Matching    │  ◀─── การจับคู่ใบสั่งซื้อ 2 ทาง (2-Way):
         │   (2-Way)        │      ทางที่ 1: Invoice → PO (จากใบแจ้งหนี้ไปหา PO)
         │                   │      ทางที่ 2: PO → Invoice (จาก PO มายังใบแจ้งหนี้)
         └────────┬─────────┘
                  ▼
         ┌──────────────────┐
         │  Error Handler   │  ◀─── จัดการข้อผิดพลาด:
         │                  │      - ข้อมูลไม่ครบ - ขอให้ OCR/LLM ประมวลผลใหม่
         │                  │      - PO ไม่พบ - แจ้งเตือน Human-in-the-Loop
         │                  │      - จำนวนเงินไม่ตรง - คำนวณ Variance %
         └────────┬─────────┘
                  ▼
         ┌──────────────────┐
         │  HITL Review     │  ◀─── ตารางตรวจสอบโดยมนุษย์ (Human-in-the-Loop)
         │  Table           │      แสดง: ผลลัพธ์ OCR, ผลลัพธ์ LLM, สถานะการจับคู่
         │                  │      พร้อมปุ่ม: ยืนยัน / แก้ไข / ปฏิเสธ
         └────────┬─────────┘
                  ▼
         ┌──────────────────┐
         │   Export CSV     │  ◀─── ส่งออกข้อมูลที่ผ่านการตรวจสอบเป็น CSV
         │                  │      พร้อมสำหรับ Import เข้าระบบ ERP
         └────────┬─────────┘
                  ▼
         ┌──────────────────┐
         │  Future ERP API  │  ◀─── (แผนอนาคต) เชื่อมต่อ API โดยตรงกับ ERP
         │  (Integration)   │      เช่น SAP, Oracle, Microsoft Dynamics
         └──────────────────┘
```

## 4. ตารางเปรียบเทียบก่อนและหลัง (Before vs After Comparison)

| หัวข้อเปรียบเทียบ | ก่อน (Manual) | หลัง (AI-Automated) | การปรับปรุง |
|------------------|---------------|---------------------|------------|
| **เวลาต่อใบแจ้งหนี้** | 20-40 นาที | 2-5 นาที | ลด 80-90% |
| **ขั้นตอนการทำงาน** | 10 ขั้นตอน (ทั้งหมดแมนนวล) | 8 ขั้นตอน (อัตโนมัติ 6 ขั้น) | ลดความซับซ้อน |
| **การตรวจจับซ้ำ** | ไม่มีระบบ (พึ่งพาความจำ) | 3-Layer Duplicate Detection | ป้องกันจ่ายซ้ำ 100% |
| **การจับคู่ PO** | ค้นหาด้วยตนเอง (5-10 นาที) | 2-Way PO Matching อัตโนมัติ | ลดเวลา 90% |
| **ข้อผิดพลาดจากการคีย์** | 2-5% ของรายการทั้งหมด | <0.5% (มี HITL ตรวจสอบ) | ลด 75-90% |
| **Audit Trail** | ไม่มี (กระดาษ/อีเมล) | ระบบบันทึกอัตโนมัติทุกขั้นตอน | ความโปร่งใสสูงสุด |
| **ความสามารถในการ Scale** | จำกัด (พนักงานมีจำกัด) | Scale ได้ไม่จำกัด (ประมวลผลคู่ขนาน) | รองรับ Volume สูง |
| **Dashboard สถานะ** | ไม่มี | มี (Real-time Status) | ความเป็นมืออาชีพ |
| **การแจ้งเตือน** | ไม่มี | อัตโนมัติทุกขั้นตอน (Email/Slack) | ตอบสนองรวดเร็ว |
| **ต้นทุนบุคลากร** | สูง (ต้องมีพนักงานเยอะ) | ต่ำ (เพิ่มประสิทธิภาพ 3-5 เท่า) | ลดต้นทุน 60-70% |

## 5. จุดตรวจสอบโดยมนุษย์ (Human-in-the-Loop Checkpoints)

ระบบ AI-Automated Workflow มีจุดตรวจสอบโดยมนุษย์ (HITL) ทั้งหมด 4 จุด เพื่อรักษาความถูกต้องและความรับผิดชอบ

```
┌──────────────────────────────────────────────────────────────────────────────────────────────┐
│                          HUMAN-IN-THE-LOOP CHECKPOINTS                                         │
│                                                                                               │
│   ┌─────────────────┐      ┌─────────────────┐      ┌─────────────────┐      ┌─────────────────┐
│   │  Checkpoint 1   │      │  Checkpoint 2   │      │  Checkpoint 3   │      │  Checkpoint 4   │
│   │  OCR Result      │      │  LLM Extraction │      │  Duplicate      │      │  Mismatch       │
│   │  Review          │      │  Review         │      │  Alert          │      │  Alert          │
│   │  (Main HITL)     │      │  (Main HITL)    │      │  Review         │      │  Review         │
│   └────────┬────────┘      └────────┬────────┘      └────────┬────────┘      └────────┬────────┘
│            │                        │                        │                        │
│            ▼                        ▼                        ▼                        ▼
│   ┌─────────────────┐      ┌─────────────────┐      ┌─────────────────┐      ┌─────────────────┐
│   │ ตรวจสอบผล OCR    │      │ ตรวจสอบข้อมูลที่  │      │ ตรวจสอบการซ้ำ   │      │ ตรวจสอบความ     │
│   │ - ความถูกต้อง    │      │ ดึงออกมาจาก LLM  │      │ - ยืนยันว่าซ้ำจริง │      │ ไม่ตรงกับ PO    │
│   │   ของข้อความ    │      │ - Invoice No.   │      │   หรือไม่        │      │ - อนุมัติเงิน    │
│   │ - ภาษา/ตัวเลข   │      │ - Date           │      │ - อนุมัติให้     │      │   แม้ไม่ตรง     │
│   │   ถูกอ่านหรือไม่ │      │ - Vendor         │      │   ข้ามได้       │      │   หรือรอ       │
│   └─────────────────┘      │ - Items[]        │      └─────────────────┘      │   จ่ายเมื่อตรง    │
│                            │ - Total          │                               └─────────────────┘
│                            └─────────────────┘
└──────────────────────────────────────────────────────────────────────────────────────────────┘
```

### รายละเอียดจุดตรวจสอบแต่ละจุด

| Checkpoint | ชื่อ | สถานะ | รายละเอียด | ผู้รับผิดชอบ |
|------------|------|-------|-----------|-------------|
| 1 | OCR Result Review | **จำเป็น** | ตรวจสอบข้อความที่ OCR อ่านออกมา ว่าถูกต้องหรือไม่ (โดยเฉพาะตัวเลขและภาษาไทย) | พนักงานบัญชี |
| 2 | LLM Extraction Review | **จำเป็น** | ตรวจสอบข้อมูล JSON ที่ LLM ดึงออกมา ยืนยัน Invoice No., วันที่, ผู้ขาย, รายการสินค้า, ยอดรวม | พนักงานบัญชี |
| 3 | Duplicate Alert Review | **แจ้งเตือน** | เมื่อระบบตรวจพบรายการซ้ำ (Layer 2 หรือ 3) พนักงานต้องยืนยันก่อนดำเนินการต่อ | หัวหน้าฝ่ายบัญชี |
| 4 | Mismatch Alert Review | **แจ้งเตือน** | เมื่อยอดใบแจ้งหนี้ไม่ตรงกับ PO เกิน Tolerance ที่กำหนด (เช่น ±5%) | หัวหน้าฝ่ายบัญชี / ผู้จัดการ |

## 6. เกณฑ์การอนุมัติตามบทบาท (Role-Based Approval Thresholds)

| บทบาท | วงเงินอนุมัติ | การตรวจสอบ | หมายเหตุ |
|------|-------------|-----------|---------|
| **พนักงานบัญชี** | ไม่มีสิทธิ์อนุมัติ | ตรวจสอบข้อมูล OCR/LLM เท่านั้น | ส่งต่อให้ผู้อนุมัติ |
| **หัวหน้าฝ่ายบัญชี** | สูงสุด 50,000 บาท | อนุมัติใบแจ้งหนี้ที่ไม่มีปัญหา | ต้องตรวจสอบ Manual หาก HITL Flag |
| **ผู้จัดการฝ่าย** | สูงสุด 500,000 บาท | อนุมัติใบแจ้งหนี้ที่มี Mismatch ไม่เกิน ±10% | ต้องมีเหตุผลประกอบ |
| **ผู้อำนวยการฝ่าย** | สูงสุด 2,000,000 บาท | อนุมัติทุกกรณี รวมถึง Mismatch ที่เกิน ±10% | ต้องลงนามอิเล็กทรอนิกส์ |
| **กรรมการบริษัท / CFO** | ไม่จำกัดวงเงิน | อนุมัติพิเศษกรณีพิเศษ | สำหรับรายการที่ต้องนำเข้าบอร์ด |

**หมายเหตุ:**
- ค่า Tolerance Default = ±5% สำหรับ Mismatch ที่ระบบอนุญาตให้ผ่านอัตโนมัติ
- รายการที่ Mismatch เกิน Tolerance จะถูก Hold ไว้รอการอนุมัติจากระดับที่สูงกว่า
- ทุกการอนุมัติต้องมี Audit Trail บันทึกในระบบ

## 7. ตารางการตัดสินใจ: ระหว่างระบบอัตโนมัติและมนุษย์ (Automation vs Human Decision Matrix)

| กรณีการตัดสินใจ | ระบบอัตโนมัติ (AI) | มนุษย์ (Human) | เหตุผล |
|----------------|-------------------|---------------|-------|
| **OCR อ่านตัวเลขถูกต้อง** | ✅ อนุมัติอัตโนมัติ | ❌ ไม่จำเป็น | ความน่าจะเป็น > 95% |
| **OCR อ่านตัวเลขผิดพลาด** | ❌ ส่งต่อ HITL | ✅ ตรวจสอบและแก้ไข | ความน่าจะเป็น < 95% |
| **ใบแจ้งหนี้ซ้ำ (Layer 1 - Exact)** | ✅ ปฏิเสธอัตโนมัติ | ⚠️ แจ้งเตือน | ตรงเงื่อนไข 100% |
| **ใบแจ้งหนี้ซ้ำ (Layer 2-3 - Fuzzy)** | ⚠️ แจ้งเตือน | ✅ ตรวจสอบและยืนยัน | อาจเป็นรายการใหม่จริง |
| **PO พบและตรงกันทุกประการ** | ✅ อนุมัติอัตโนมัติ | ❌ ไม่จำเป็น | Match 100% |
| **PO ไม่พบ** | ❌ ส่งต่อ HITL | ✅ ค้นหาหรือสร้าง PO ใหม่ | ต้องใช้ดุลยพินิจมนุษย์ |
| **ยอด Mismatch < Tolerance (±5%)** | ✅ อนุมัติอัตโนมัติ | ❌ ไม่จำเป็น | อยู่ในช่วงยอมรับได้ |
| **ยอด Mismatch > Tolerance (±5%)** | ❌ ส่งต่อ HITL | ✅ ตรวจสอบและอนุมัติ | ต้องมีเอกสารประกอบ |
| **ข้อมูลผู้ขายไม่ตรงกับระบบ** | ❌ ส่งต่อ HITL | ✅ ตรวจสอบ Vendor Master | อาจเป็นผู้ขายใหม่ |
| **วันที่ใบแจ้งหนี้เก่ากว่า 90 วัน** | ⚠️ แจ้งเตือน | ✅ ตรวจสอบ Statute of Limitations | อาจมีปัญหาทางกฎหมาย |
| **จำนวนรายการในใบแจ้งหนี้ > 100 รายการ** | ✅ ประมวลผลต่อ | ⚠️ สุ่มตรวจ 10% | ประสิทธิภาพมวลรวม |
| **ใบแจ้งหนี้จากผู้ขายใหม่ (ไม่มีในระบบ)** | ❌ ส่งต่อ HITL | ✅ สร้าง Vendor ใหม่ | ต้องมีการอนุมัติ |

## สรุป

ระบบ Automated Invoice Verification ใช้ AI ช่วยในการประมวลผลใบแจ้งหนี้ ลดเวลาจาก 20-40 นาที เหลือ 2-5 นาทีต่อรายการ (ลดลง 80-90%) โดยยังคง **จุดตรวจสอบโดยมนุษย์ (HITL) 4 จุด** เพื่อรักษาความถูกต้องและความรับผิดชอบ พร้อมระบบ **เกณฑ์การอนุมัติตามบทบาท** ที่ชัดเจน และ **ตารางการตัดสินใจ** ที่โปร่งใส ทำให้องค์กรสามารถ Scale การทำงานได้โดยไม่ต้องเพิ่มบุคลากรในสัดส่วนเท่ากัน

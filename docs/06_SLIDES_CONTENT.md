## สไลด์ที่ 4: กรณีศึกษาตัวอย่าง (Case Study)

### Case 1: การจ่ายเงินผิด 5,000 บาท

```
สถานการณ์:
  พนักงานตรวจสอบ Invoice จาก บริษัท วัสดุก่อสร้าง จำกัด
  Invoice ระบุยอด 12,000 บาท
  PO อนุมัติไว้ 7,000 บาท
  พนักงานไม่ทันสังเกตว่ายอดไม่ตรง → อนุมัติจ่ายเกิน 5,000 บาท

ผลกระทบ:
  - สูญเสียเงิน 5,000 บาท
  - ต้องดำเนินการขอคืนเงินจากผู้จัดจำหน่าย
  - ใช้เวลาแก้ไข 1–2 วันทำการ
```

### Case 2: การจ่ายเงินซ้ำ (Duplicate Payment)

```
สถานการณ์:
  แผนกบัญชีได้รับ Invoice เดียวกัน 2 ครั้งจากผู้จัดจำหน่าย
  (ส่ง Email ซ้ำโดยไม่ตั้งใจ)
  ไม่มีระบบตรวจจับ → จ่ายเงินค่าสินค้าเดียวกัน 2 ครั้ง

ผลกระทบ:
  - จ่ายเกิน 100% ของมูลค่าสินค้า
  - ต้องติดตามขอเงินคืน
  - เสียความน่าเชื่อถือกับผู้สอบบัญชี
```

## สไลด์ที่ 8: Technology Stack

| ชั้น (Layer) | เทคโนโลยี | หมายเหตุ |
|-------------|----------|---------|
| **Frontend / UI** | Streamlit (Python Web App) | เป็นมิตรต่อผู้ใช้ที่ไม่มีความรู้ด้านเทคนิค |
| **OCR Engine** | Typhoon OCR (scb10x/typhoon-ocr1.5-3b) | รองรับภาษาไทยโดยเฉพาะ |
| **LLM Engine** | Typhoon LLM (typhoon-v2.5-30b-a3b-instruct) | สกัด Structured JSON จาก Raw Text |
| **Orchestration** | Python OrchestratorAgent | Swarm Multi-Agent Pattern |
| **Data Storage** | JSON Files (Flat File) | po_vendor_data.json, paid_invoices.json, audit_logs.json |
| **Runtime Mode 1** | LOCAL — Ollama | ใช้งานฟรี, ไม่ต้อง API Key, ต้องมี GPU |
| **Runtime Mode 2** | API — Typhoon Cloud | แม่นยำกว่า, มี Rate Limit 20 req/min |
| **Notification** | LINE Messaging API | แจ้งเตือนเมื่อพบปัญหา |
| **Deployment** | Local Machine / Cloud Server | Python 3.11+, ไม่ต้องติดตั้งเพิ่ม |

## สไลด์ที่ 9: คุณสมบัติหลัก (Key Features)

| # | คุณสมบัติ | รายละเอียด |
|---|---------|-----------|
| 1 | **Batch OCR Processing** | รองรับ PDF, JPG, PNG หลายไฟล์พร้อมกัน |
| 2 | **LLM Structured Extraction** | แปลง Raw Text → JSON (Invoice Number, Vendor, Tax ID, Amount, PO Ref, VAT) |
| 3 | **3-Layer Duplicate Detection** | Exact Match → Fingerprint (Hash) → Fuzzy Match (Similarity ≥ 0.85) |
| 4 | **2-Way PO Matching** | เทียบ PO Number + Amount → MATCHED / AMOUNT_MISMATCH / NOT_FOUND |
| 5 | **Human-in-the-Loop (HITL)** | Staff ตรวจสอบ แก้ไข และ Approve ทุกใบก่อนจ่ายเงิน |
| 6 | **Thai VAT Auto-calculation** | คำนวณ VAT 7% อัตโนมัติเมื่อ LLM คืน vat=0 |
| 7 | **Dual-Mode LOCAL/CLOUD** | สลับระหว่าง Ollama (ฟรี) กับ Typhoon API (แม่นยำ) ได้ |
| 8 | **Admin Panel** | CRUD PO/Vendor Database, ดู/ลบ Paid Invoice, ดู Audit Log |
| 9 | **CSV Export with Audit** | ดาวน์โหลด CSV (UTF-8-sig) พร้อมข้อมูลผู้ตรวจสอบและเวลา |
| 10 | **Error Recovery** | Retry 3 ครั้ง + Rate Limit Cooldown + LOCAL Fallback |

## สไลด์ที่ 10: Demo Scenario 1 — การไหลปกติ: MATCHED Invoice

### ขั้นตอนการทำงาน (Flow)

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  สถานการณ์: Invoice มาถึง → ระบบประมวลผล → MATCHED → Staff Approve         │
│                                                                             │
│  1. OCR AGENT                                                                │
│     📄 Invoice (PDF/JPG) ──→ Typhoon OCR ──→ Raw Thai/English Text         │
│                                                                             │
│  2. EXTRACTION AGENT (LLM)                                                  │
│     📝 Raw Text ──→ Typhoon LLM ──→ Structured JSON:                       │
│        {                                                                     │
│          "invoice_number": "INV-2024-001",                                  │
│          "vendor_name": "บริษัท วัสดุก่อสร้าง จำกัด",                          │
│          "po_reference": "PO-2023-001",                                     │
│          "total_amount": 10700.00                                          │
│        }                                                                     │
│                                                                             │
│  3. DUPLICATE DETECTION AGENT (Layer 1-3)                                    │
│     ✓ Layer 1: Exact Match — ไม่พบ INV-2024-001 ใน paid_invoices          │
│     ✓ Layer 2: Fingerprint Match — Hash ไม่ตรงกับ Cache                    │
│     ✓ Layer 3: Fuzzy Match — Similarity < 0.85                             │
│     → Result: NOT DUPLICATE                                                 │
│                                                                             │
│  4. PO MATCHING AGENT                                                        │
│     ✓ PO-2023-001 พบในระบบ                                                  │
│     ✓ Invoice Amount: 10,700 บาท = PO Approved: 10,700 บาท                │
│     → Result: ✅ MATCHED                                                    │
│                                                                             │
│  5. HITL INTERFACE                                                          │
│     📋 พนักงานเห็นผล: MATCHED → [✅ Approve] หรือ [❌ Reject]                │
│                                                                             │
│  6. APPROVE → บันทึกเข้า paid_invoices.json + Audit Log + Export CSV      │
└─────────────────────────────────────────────────────────────────────────────┘
```

## สไลด์ที่ 11: Demo Scenario 2 — Duplicate DETECTED Alert

### ขั้นตอนการทำงาน (Flow)

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  สถานการณ์: Invoice ซ้ำ — ระบบตรวจจับได้ 3 ชั้น → Alert → Reject             │
│                                                                             │
│  1. OCR AGENT                                                                │
│     📄 Invoice (INV-0001) ──→ Typhoon OCR ──→ Raw Text                     │
│                                                                             │
│  2. EXTRACTION AGENT (LLM)                                                  │
│     { "invoice_number": "INV-0001", "total_amount": 5000.00 }              │
│                                                                             │
│  3. DUPLICATE DETECTION AGENT — 3-Layer Check                               │
│                                                                             │
│     ┌─────────────────────────────────────────────────────────────────────┐  │
│     │ 🔍 Layer 1: Exact Match                                            │  │
│     │    เทียบ invoice_number ตรงกันทุกตัวอักษร (case-insensitive)        │  │
│     │    → INV-0001 พบใน paid_invoices.json ✅                           │  │
│     └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│     ┌─────────────────────────────────────────────────────────────────────┐  │
│     │ 🔍 Layer 2: Fingerprint Match (ถ้า Layer 1 ไม่พบ)                  │  │
│     │    Hash = SHA256(vendor_name + total_amount + invoice_date)       │  │
│     │    เปรียบเทียบกับ Cache                                             │  │
│     └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│     ┌─────────────────────────────────────────────────────────────────────┐  │
│     │ 🔍 Layer 3: Fuzzy Match (ถ้า Layer 1-2 ไม่พบ)                      │  │
│     │    SequenceMatcher(s1, s2) ≥ 0.85                                 │  │
│     │    จับคู่ invoice_number ที่คล้ายกัน เช่น INV-0001 ≈ IV1806-0001   │  │
│     └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│  → Result: 🚨 DUPLICATE_INVOICE DETECTED                                   │
│                                                                             │
│  4. HITL INTERFACE                                                          │
│     📋 พนักงานเห็นผล: 🚨 DUPLICATE → [❌ Reject]                             │
│                                                                             │
│  5. LINE NOTIFICATION                                                       │
│     📱 แจ้งเตือนหัวหน้า: "พบใบแจ้งหนี้ซ้ำ INV-0001 ยอด 5,000 บาท"             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## สไลด์ที่ 14: ทำไมต้อง Multi-Agent ไม่ใช่ Single Agent?

### เหตุผลที่ใช้ Swarm Multi-Agent Architecture

| ด้าน | Single Agent | Multi-Agent Swarm |
|------|-------------|-------------------|
| **Separation of Concerns** | ทำทุกอย่างใน Prompt เดียว → Prompt ยาวมาก, LLM สับสน | แต่ละ Agent มีหน้าที่ชัดเจน → Prompt กระชับ |
| **Error Isolation** | ถ้าตรวจสอบผิด → ทั้งระบบหยุด | Agent ใดล้มเหลว → Retry เฉพาะส่วน |
| **Parallel Processing** | ประมวลผลทีละขั้นตอนทั้งหมด | Agent ต่างๆ ทำงานเป็นลำดับหรือขนานได้ |
| **Debugging** | ยากตรวจสอบว่าขั้นตอนไหนผิด | ดู Log เฉพาะ Agent ที่ล้มเหลว |
| **Maintainability** | แก้ไข Logic ต้องแก้ Prompt หลักทั้งหมด | เพิ่ม/แก้ไข Agent เฉพาะส่วนได้ |
| **Specialization** | โมเดลเดียวทำ OCR + Extraction + Matching | แต่ละ Agent ใช้ Prompt ที่เหมาะกับงาน |
| **Scalability** | เพิ่มงาน → Prompt บวม → ค่าใช้จ่ายสูง | เพิ่ม Agent ใหม่โดยไม่กระทบส่วนอื่น |

### หลักการออกแบบ
```
Single Agent (Monolithic)
┌──────────────────────────────────────────────────────┐
│  1 Prompt ใหญ่มาก: "อ่าน Invoice → สกัดข้อมูล →      │
│   ตรวจซ้ำ → จับคู่ PO → ตัดสินใจ..."                  │
│  → ❌ LLM สับสน, Prompt ยาว, ค่าใช้จ่ายสูง              │
└──────────────────────────────────────────────────────┘

Multi-Agent Swarm (Pipeline)
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  OCR     │ →  │Extract(LLM)│ → │ Duplicate│ →  │PO Match  │
│  Agent   │    │  Agent    │    │Detection │    │  Agent   │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
   ~50 lines      ~30 lines       ~40 lines      ~30 lines
  → ✅ แต่ละส่วนกระชับ, ง่ายต่อการ Debug และปรับปรุง
```

## สไลด์ที่ 18: LOCAL vs CLOUD — Dual Mode Comparison

| ด้าน | LOCAL (Ollama) | API (Typhoon Cloud) |
|------|---------------|---------------------|
| **การติดตั้ง** | ต้องติดตั้ง Ollama + ดาวน์โหลดโมเดล (~6–10 GB) | ไม่ต้องติดตั้งเพิ่ม เพียงมี API Key |
| **ค่าใช้จ่าย** | ฟรี (ใช้เครื่องตัวเอง) | จ่ายตามการใช้งานจริง (Token-based) |
| **Rate Limit** | ไม่มี (ขึ้นกับ Hardware) | 20 req/min (Typhoon Cloud) |
| **ความเร็ว** | เร็วถ้ามี GPU แรง (RTX 3080+) | ~2–5 วินาที/ใบ (รวม Latency) |
| **ความแม่นยำ** | ดี (เทียบเท่า Cloud) | ดีที่สุด (โมเดลล่าสุด) |
| **ความเสถียร** | ขึ้นกับเครื่อง | สูง (Dedicated Infrastructure) |
| **ดาต้าร์ต้อง** | GPU VRAM ≥ 8 GB (แนะนำ 16 GB) | เพียงแค่ Internet |
| **เหมาะกับ** | Development, Demo, องค์กรที่มี GPU | Production, ผู้ใช้ทั่วไป |
| **ข้อมูลส่วนตัว** | ข้อมูลไม่ออกนอกเครื่อง (✅ ปลอดภัย) | ข้อมูลส่งไป Cloud (⚠️ ต้องพิจารณา) |
| **OCR Model** | scb10x/typhoon-ocr1.5-3b | typhoon-ocr |
| **LLM Model** | typhoon-v2.5-30b-a3b-instruct | typhoon-v2.5-30b-a3b-instruct |

### สรุป: เลือกใช้ตามสถานการณ์
- **Demo/Development** → LOCAL (Ollama) ฟรี, ไม่ต้องรอ
- **Production** → API (Cloud) แม่นยำ, เสถียร
- **Hybrid** → ระบบสลับโหมดได้ทันที (Streamlit Sidebar)
